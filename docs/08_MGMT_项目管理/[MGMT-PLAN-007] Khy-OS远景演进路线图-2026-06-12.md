## 0. 北极星与当前坐标

### 0.1 北极星（三年终局）
> **用户开机，直接进入 Khy-OS；AI 是系统级能力而非外挂应用；从内核到 agent 垂直整合、无供应商锁定；
> 平台能长出第三方应用并形成生态抽成。**

终局可证伪的三个硬指标：
1. **可日用**：在真实硬件（或稳定虚机）上自托管运行 ≥ 8 小时不崩，能编辑代码、跑 agent、联网。
2. **AI 原生**：内核内常驻 agent 能在无外部网络兜底时，自主完成「读—改—验—修」闭环至少一类任务。
3. **生态**：≥ 3 个由他人开发、经 `khyos.apps` 协议接入的应用稳定运行，其中 ≥ 1 个走付费/抽成。

### 0.2 当前坐标（2026-06-12 实测基线）
| 维度 | 已落地 | 成熟度 |
|---|---|---|
| 内核 | 引导/分页COW/进程fork-exec-wait/抢占调度/syscall/IPC/VFS+磁盘持久化/信号/管道/ELF+PE 双格式/窗口管理/MoonBit 内嵌；阶段 1–9 全 QEMU 实测 | **QEMU 可用，未自托管** |
| Agent⇄OS | AgentBus A1–A8（控制面/决策面/事件面，COM2+COBS+CRC16）+ host 桥 + MCP 注册 + 内置 brain 接网关 | **旁路协同闭合，未接管调度** |
| AI Shell/网关 | CLI REPL + 16 适配器级联网关 + 104 工具 + 工作流引擎 + TUI（ink）+ 多租户网关 | **Claude-Code 级，工业化进行中** |
| 约束/自愈 | 元帅双模式（Goal10）+ 元规划/动态约束注入/信任熔断/宪法红线（Goal11） | **协议层完成，未接管 executeTool** |
| 应用生态 | `app_protocol.py` 单一真源 + entry_points 发现/懒激活 + 路径主权隔离；khyquant 首款应用 | **协议就绪，无第三方应用、无市场** |
| 发行 | pip/npm 双渠道孪生 + 双阶段自愈 devenv + `.ai/` 种子文档 + 项目元数据 pre-commit | **可一键装，无开机即用 ISO** |
| 商业化 | 许可变现可执行 + 多租户网关 SaaS 雏形 | **路径清晰，未规模化** |

**一句话**：协议与地基已厚，**三道"最后一公里"未通**——内核未自托管、agent 未接管执行、生态无第三方。
本路线图即围绕打通这三道关组织。

## 2. 主线 A — 内核：从 QEMU 玩具到可自托管日用 OS

> 红线（不可违背）：**"内核绝不乱动，沙箱任你卷"**——核心微/宏混合层用 C，可替换层用 MoonBit WASM；
> 每一步必须保留 QEMU 一键复现 + 回归绿。落点全部在 `kernel/`。

### A1. 块设备持久化深化（M1，关键路径）
- **现状**：`ata.c`+`persist.c`+`diskfs.c` 已挂 `/disk`，但持久化深度未经压力验证。
- **细节**：
  - `diskfs.c` 增加 inode 位图 + 块分配器的崩溃一致性（写前日志或双写 superblock）。
  - `ata.c` 支持 LBA48 + 多扇区 DMA（当前若为 PIO 单扇区，吞吐瓶颈）。
  - 落 `fsck` 式启动自检：挂载前校验 superblock magic + 块计数。
- **验收**：QEMU 内创建 100 个文件、各写 4KB、`sync`、重启，全部内容逐字一致；断电模拟（QEMU `quit` 写中途）后启动能自检并丢弃半写块而非挂死。

### A2. 真实键盘 stdin 全链路（M1→M2）
- **现状**：`keyboard.c` 驱动 + shell 行编辑可用，但进程级 `read(fd0)` 阻塞读真实键盘的全链路待打通。
- **细节**：键盘中断 → 行规程缓冲（canonical/raw 两态）→ `tty` 设备 → `vfs` 的 `/dev/tty` → 进程 `read(0)` 唤醒。复用 `ipc.c` 的 lost-wakeup-safe 阻塞范式。
- **验收**：内核内运行一个读 `stdin` 的 C 程序（如 `cat`），键盘输入实时回显并被进程读到；Ctrl-C 经信号链 `sigaction` 中断。

### A3. 内核内自托管工具链（M2，里程碑核心）
- **目标**：内核内能「编辑 → 编译 → 运行」一个 C 程序，迈出"自托管"第一步。
- **细节**：
  - 移植一个极简编辑器（或强化 `shell.c` 的行编辑为多行）。
  - 评估两条路：①移植 `tcc`（Tiny C Compiler，单文件、ELF 输出，最契合）；②MoonBit→WASM→内核 WASM 运行时。优先 ①验证可行性。
  - `exec` 已支持 ELF，编译产物可直接 `exec`。
- **验收**：内核 shell 里 `edit hello.c` → `cc hello.c -o hello` → `./hello` 打印 "hello"，全程不出 QEMU。
- **风险**：tcc 依赖的 libc 子集需裁剪移植，工作量大；可先支持 freestanding 子集。

### A4. 内存/调度健壮化（贯穿 M1–M2）
- **细节**：`vmm.c` 的 COW 缺页加引用计数回收审计；`sched.c` 加优先级老化防饥饿；`kheap.c` 加 use-after-free 红区探测（debug 构建）。
- **验收**：跑 `fork` 炸弹（受限）+ 长时调度压力 1h，无页泄漏（pmm 空闲页计数回到基线 ±阈值）、无死锁。

### A5. 网络栈实用化（M3）
- **现状**：`net.c` 存在但能力浅。
- **细节**：补 ARP/ICMP/UDP→最小 TCP；目标是内核内 agent 能发一次 HTTP 请求兜底（与决策面互补）。
- **验收**：内核内 `ping` 网关通；UDP echo 通；TCP 能完成一次短连接 GET。

### A6. 裸机/稳定虚机自托管（M4，终局）
- **细节**：真实硬件驱动补齐（AHCI/NVMe 之一、USB 键盘、真实显卡 framebuffer via GOP）；或锁定一个稳定虚机配置（QEMU/KVM 固定机型）作为"官方日用环境"。
- **验收**：在目标环境自托管运行 8h，期间编辑代码 + 跑 agent + 联网不崩。
- **取舍**：真机驱动是无底洞；**单人现实下优先"稳定虚机日用"**，真机作为长尾彩蛋。

## 3. 主线 B — Agent⇄OS 协同：从旁路桥到内核级自主体

> 已完成 A1–A8 八阶段（控制面/决策面/事件面三面闭合）。下一程是**让 agent 从"旁观+被问"走向"参与内核决策与自愈"**。落点 `kernel/`（agentbus 族）+ `kernel/bridge/`。

### B1. 决策面真接内核决策点（M1，关键路径）
- **现状**：A5 已实现 `SYSCALL_AGENT_ASK`（OS→agent 提问，3s 兜底永不挂死），但内核中实际"问 agent 再决策"的决策点仅示例级。
- **细节**：选 1–2 个**安全可逆**的内核决策点接入决策面，例如：
  - 缺页时的换页策略提示（agent 给 hint，内核保留兜底）；
  - shell 未知命令的智能纠错建议（agent 返建议，用户确认）。
  - 严守："agent 只给 hint，内核永远有确定性兜底，超时即兜底"——绝不让 agent 成为内核存活的必要条件。
- **验收**：拔掉 agent（bridge 不连），内核所有决策点走兜底正常运行；接上 agent，决策点采纳建议且可观测。

### B2. 内核内 agent 常驻雏形（M2→M3）
- **细节**：让 host 桥的内置 brain（`khy-brain-gateway.js`）能以"内核内常驻任务"形态运行一个轻量决策循环（而非纯外部进程）；用 MoonBit WASM 在内核内跑一个极小的策略核，重决策仍出 COM2 问外部网关。
- **验收**：QEMU 内 agent 任务常驻，事件面（SPAWN/EXIT/FAULT）触发它做出一次记录/建议动作。

### B3. 故障自愈闭环（M3→M4，终局能力）
- **细节**：事件面 FAULT → 内核内 agent 捕获 → 经决策面/工具面读取故障进程上下文 → 提出修复（重启/隔离/打补丁建议）→ 控制面执行可逆动作。这正是北极星指标 ②"读—改—验—修"的内核版。
- **验收**：构造一个会 fault 的进程，agent 自动隔离它并产出一条人类可读的根因+建议，全程不崩内核。
- **依赖**：B1（决策面接入）+ 主线 D（约束保证 agent 的修复动作受元规划/红线约束）。

### B4. 协议加固与安全（贯穿）
- **细节**：控制面写操作的 `/bin` 越权 EPERM 守卫已有；补：决策面/工具面的能力令牌（capability，复用 `cap_init`/`capability.c`）+ 帧重放防护（序号+nonce）。
- **验收**：伪造帧/重放帧被拒；越权写被 EPERM；模糊测试 1h 不崩。

## 4. 主线 C — AI Shell 与网关：从 CLI 到工业级 agentic 运行时

> 这是**当前核心卖点**，也是营收最近的一条线。落点 `services/backend/`。

### C1. 元规划接管真实执行（M1，与主线 D 共享关键路径）
- **现状**：Goal11 的 `MetaPlanCoordinator` 协议层完成，但**未接管 `executeTool`**（toolCalling.js:2342）。
- **细节**：在 `executeTool` 前置一道可开关的钩子（`KHY_METAPLAN_ENFORCE`），对**写类工具**（editFile/writeFile/createDocument/executeCode/runShell）按元规划流程跑：渲染元规划 prompt → 模型产出 → ingest → validateExecution → 执行 → recordOutcome。读类工具与低风险默认快放行。
- **验收**：改注释走 Soft 极速；重构走 Hard 真 babel 校验；删库被宪法红线强制 Block；连续 2 次 Soft 翻车后同会话强制 Hard——全部在真实工具链上复现，且开关关闭时行为与今天完全一致（零回归）。
- **风险**：每次写都让模型先产元规划会增加一轮往返；故默认只对"风险工具 + 非平凡改动"启用，平凡改动走启发式直通。

### C2. Hooks 系统（M2）
- **细节**：`PreToolUse`/`PostToolUse`/`PrePrompt`/`PostResponse` 四事件，声明式 `hooks.json`；与已有 `hookRunner`/守卫审批链整合（记忆中 guardApproval 已铺垫）。
- **验收**：用户配一条 PreToolUse 钩子拦截 `rm -rf` 并改写为回收站移动，实测生效。

### C3. PTY + Headless/SDK 模式（M2→M3）
- **细节**：`node-pty` 伪终端支持交互式/sudo 命令；JSON stdin/stdout 非交互协议供 CI、IDE 后端、ACP 对接。Python SDK 与 Node SDK 双绑定（记忆中 SDK v1.1.0 已起步）。
- **验收**：外部进程经 headless 协议驱动一次完整 agent 任务并取回结构化结果；PTY 下 `top`/`vim` 可交互。

### C4. 网关韧性与成本（贯穿）
- **细节**：16 适配器的健康度自动巡检 + 模型能力分级（modelTier 已有）按任务路由最省钱够用的后端；Token 优化（Schema/Response 压缩，对标 Tokenless）。
- **验收**：同一批任务在开启智能路由后总 Token/成本下降可量化（看板出数）。

### C5. 可观测性深化（M3）
- **细节**：工具调用、模型推理、适配器路由、状态流转的**全程透明**是差异化卖点（"教练车"）。补语义事件流 + 可回放的 trace 时间线（traceAudit 已有地基）。
- **验收**：任一次 agent 任务可在 UI 上逐步回放：每个工具的入参/出参/耗时/路由决策/约束策略一目了然。

## 5. 主线 D — 约束与自愈：元帅 + 元规划真正接管

> Goal10（元帅）+ Goal11（元规划）已建协议，本主线让它们**从"旁置子系统"变成"执行路径的实际守门人"**。落点 `services/backend/src/services/{marshal,metaplan}/` + 调度器接线。

### D1. 元规划执行接管（= C1，M1）
见 C1。这是 D 主线的入口。

### D2. 元帅接管宏观规划（M2）
- **细节**：把 `MarshalCoordinator` 接到 Goal Mode / 大任务编排的"主控选择"上：用户 `--marshal` 指定或自动选举的元帅，其产出的计划经元规划逐微步落地；弱主走选择题+规则引擎兜底，强主放权。
- **验收**：用一个弱模型当元帅跑一个多步任务，全程不产生不可逆破坏、计划逻辑自洽；强模型当元帅则自由规划。

### D3. System_Block 接备份引擎（M2）
- **现状**：Goal11 的 `System_Block` 以 `requireSnapshot/requireConfirmation` 暴露契约，未绑实际快照。
- **细节**：绑定 `rollbackService`（记忆中已存在）：Block 级动作执行前自动打工作区快照，失败可一键回滚；复用主线 F 的 ws-ckpt（工作区快照）能力。
- **验收**：一个 Block 级危险动作执行前自动快照，人工拒绝则零副作用，执行后可回滚到快照点。

### D4. 信任熔断的跨会话学习（M3）
- **细节**：当前熔断是会话级；升级为可选的跨会话持久化"风险画像"（某类操作历史翻车率高→默认提高约束级别），写入 `~/.khyos`。
- **验收**：历史上常翻车的操作类型，新会话默认起步即 Code_Hard，可被用户重置。

### D5. 安全内核对标（M3，吸收 MGMT-PLAN-006 的 Agent Sec Core）
- **细节**：Prompt 注入检测（规则+语义+ML 三层）、代码扫描引擎、技能签名验证——作为元规划"宪法红线"的扩展规则源。
- **验收**：已知 Prompt 注入样本被拦截；危险代码块在写入前被代码扫描标记。

## 6. 主线 E — 应用生态：从协议到市场与抽成

> 架构已解耦（`app_protocol.py` 单一真源、路径主权、entry_points 发现/懒激活）。本主线把"能接应用"变成"有应用、能赚钱"。落点 `platform/khy_platform/` + 新增运行时/市场。

### E1. WASM 应用运行时（M3，里程碑核心）
- **细节**：MoonBit→WASM 作为跨平台应用首选交付格式；后端起一个 WASM 沙箱执行器（wasmtime/wasmer 绑定，记忆中 wasmAppService 已有雏形），暴露受限系统能力（文件走应用主权目录、网络走公共 API）。
- **验收**：一个 MoonBit 写的小应用编译为 WASM，经 `khyos.apps` 协议接入，在沙箱内跑通且无法越权访问底座数据。

### E2. 第二/第三款应用（M3→M4，终局指标）
- **细节**：自研或引导他人开发 ≥ 2 个非 khyquant 应用（如：笔记/知识库 agent、本地自动化 agent），验证协议的通用性；沉淀"应用开发脚手架"（`khy create-app`）。
- **验收**：≥ 3 个应用（含 khyquant）经协议稳定共存，互不污染数据主权。

### E3. 应用市场雏形 + 抽成（M4）
- **细节**：`khy apps search/install/publish`；签名与版本；付费应用走计费（复用多租户网关的计量能力）抽成。
- **验收**：一笔付费应用安装→计费→抽成全链路打通（哪怕测试金额）。

### E4. khyquant 作为旗舰示范（贯穿）
- **细节**：持续把 khyquant 打磨成"协议能力的活样板"（双模入口、路径隔离、Android APK 已通），它的每个新能力都反哺协议。
- **验收**：khyquant 不再依赖任何底座内部实现，仅经公共 API/协议——作为解耦的反向证明。

## 7. 主线 F — 发行 / 可维护 / 社区：从 pip 到开机即用

### F1. 开机即用 ISO（M4，终局载体）

## 11. 依赖关系与关键路径

```
关键路径（决定终局节奏的最长链）：
  A1 块设备持久化 ─┐
                  ├─→ A3 内核自托管工具链 ─→ A6 自托管日用 ─┐

                                                           ├─→ M4 开机即用

                                                           │
```

- **最关键单点**：`A3 内核自托管工具链`（tcc 移植）与 `C1/D1 元规划接管 executeTool`——前者决定"内核是不是真 OS"，后者决定"agent 是不是真受控"。两者都已有清晰落点，建议**优先 C1/D1**（投入产出比最高、立即提升卖点），A3 作为 M2 攻坚。
- **可并行**：主线 E（生态）、F3（GitHub 发布）、G（商业化 P0/P1）与内核线无强依赖，可随时推进换取现金流与曝光。

## 12. 风险登记与单人开发取舍

| 风险 | 级别 | 应对 |
|---|---|---|
| 真机驱动是无底洞 | 高 | M4 优先"稳定虚机日用"，真机作长尾；不把终局押在真机 |
| tcc/自托管工具链移植量大 | 高 | 先 freestanding 子集；失败则退守 WASM 运行时路线 |
| 元规划接管增加往返延迟 | 中 | 默认只守"风险工具+非平凡改动"，平凡改动启发式直通；开关可关 |
| 单人精力分散于七主线 | 高 | 严守关键路径优先级（§11）；生态/商业化线换现金流反哺 |
| 内核回归脆弱 | 中 | 每步保留 QEMU 一键复现 + 接管路径"开关可关=同今天" |
| 生态冷启动（无第三方） | 中 | 自研第 2/3 应用打样 + `khy create-app` 脚手架降低门槛 |
| 安全面扩大（agent 入内核） | 高 | capability 令牌 + 宪法红线 + agent 永非内核存活必要条件 |

**取舍总纲**：内核线求"够用日用"而非"功能齐全"；agent 线求"受控自主"而非"全自动"；
生态/商业线求"先有一笔/一个"而非"先有规模"。**宁可窄而通，不要宽而虚。**
