# 弱模型专属护栏（在 Khy-OS 上稳扎稳打）

如果你是较小/较弱的模型，或任务复杂、上下文吃紧——照这份做，能少犯错、不半途崩。

## 核心心法：一次只做一步

- **把大任务拆成小步**，一步只改一个文件、只做一件事。做完一步、确认无误，再下一步。
- 每步动手前用一句话说清"这一步要干什么"，做完用一句话说"这一步结果"。
- 不确定就**先只读**（读文件、grep、体检），别急着写。

## 上下文与截断（小模型最容易栽的地方）

- **别让工具输出撑爆上下文**：读大文件用行范围（`offset/limit`）；`grep` 加过滤别拉全仓；`git diff` 限定文件。
- **怀疑被截断**：如果你的回复像是中途断了（没有结论、代码没闭合），**明说"我可能被截断了"**，然后分多次输出，别假装完整。
- 一次输出别塞太多代码；长内容分块给。

## 工具调用（照约定，别自创格式）

- 工具调用参数是 **JSON**，**不要把 JSON 包在 ```代码块``` 里**——直接产结构化调用。
- 参数名照工具定义写，别猜别改名。
- 有的 provider 不接受 `role:'tool'` 消息——**别去破坏 khy 已有的兼容层**，它会帮你转换；你只管正常调用工具。
- 一次一个工具调用更稳；没把握就别并发一堆。

## 改代码时（务必配合 `/khy-safe-change`）

- **只加不改**：新逻辑写成纯叶子 + `KHY_` 门控（默认开）+ 关掉逐字节回退。
- **一次只碰一个文件**。要改多个文件时，一个一个来，每个都能单独回退。
- 改完**立刻**：跑相关单测、看守卫是否绿。红了就地修，别往下堆。
- 拿不准某函数/文件在不在 → **先 grep/读确认**，别凭记忆引用（防幻觉）。

## 卡住了怎么办

- 卡住别硬编。说清"卡在哪、试了什么、需要什么"，请用户决定，或激活 `/khy-troubleshoot` 分类排错。
- 每轮最多问一个澄清问题，且先给出你能给的答案再问。

## 收尾

做完激活 `/khy-honest-closure`：如实说改了什么、接到哪、测了几个绿、有没有没做的。**跑了工具 ≠ 做完了。**
