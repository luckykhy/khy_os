# 🛟 命脉自保清单（给你自己看）

> 你**怕把项目搞坏、要发布、或要回退**时看这篇。全部是你自己敲的命令或双击的入口，
> **不依赖 AI**。命令都尽量用完整路径（最抗坏）。更完整版见 `docs/传承/KHY-OS-传承书.md`。

---

## 项目靠这四根支柱活着（心里有数就行）

| 支柱 | 是什么 | 没了会怎样 |
| --- | --- | --- |
| **pip 生命线** | 唯一的下载与发布渠道（打成 wheel 上传 PyPI） | 无法分发，等于消失 |
| **守卫免疫系统** | 提交时自动拦截"越改越差"的机器门禁 | 弱模型/手滑会慢慢腐蚀代码 |
| **`.ai/` 种子文档** | 机器可读的项目地图与红线 | 新人/新 AI 看不懂结构 |
| **`maintenance/` 双击 + `khy maintain`** | 不懂命令行也能维护的图形入口 | 非技术维护者无从下手 |

## 我怕改坏 · 先做只读体检（不会改任何东西）

```bash
node services/backend/bin/khy.js maintain freshness   # 与时俱进体检（只读）
node services/backend/bin/khy.js maintain             # 维护者驾驶舱（只读）
node services/backend/bin/khy.js health               # 顶层自助诊断
```

**不会敲命令**？双击 `maintenance/` 里名字排最前的 `维护-0-从这里开始`
（macOS `.command` / Windows `windows\*.bat` / Linux `linux/*.sh`），按编号选。

常用双击入口：

| 想干什么 | 双击这个 |
| --- | --- |
| 看现在健不健康 | `维护-一键健康体检` |
| 升级前先备份 | `维护-备份关键数据配置` |
| 升级坏了要回滚 | `维护-回滚到最近稳定版本` |
| 依赖坏了重装 | `维护-一键重装依赖` |
| 求助前导出诊断 | `维护-导出诊断日志` |

> 🆘 **慌乱中只想止血**？翻 `docs/传承/紧急恢复卡片.md`——一页纸，每种"坏了"都给两条路。

## AI 改完了 · 我该验什么（发布前的最小闭环）

按顺序，任何一步红就别发布：

```bash
# 1) 只读体检无红
node services/backend/bin/khy.js maintain freshness

# 2) 守卫全绿（做一次提交就会自动触发 pre-commit 钩子；红了先修）
git add -A && git commit -m "..."     # 钩子会自动跑 scripts/check-*.js

# 3) 上帝组件没恶化
npm --prefix services/backend run arch:god
```

**盯死这三个问题问 AI**（它答不好就别信"做完了"）：
1. 「这改动接到 executeTool / toolUseLoop / aiManagementServer 哪个入口了？」
2. 「把那个 KHY_ 开关设成 off，行为和改之前一样吗？」
3. 「守卫全绿吗？没绕过对吧？」

## 我要发布到 pip · 照手册做

**权威手册**：`docs/07_OPS_运维/[OPS-MAN-042] 发布手册-pip与npm-无AI照做.md`（无 AI 也能照做）。

要记住的两件事：
1. **改了仓库源码，pip 用户拿不到**，除非重新打包发版。
2. **版本号要多处一致**——别只改一处（历史上出过 backend package.json 与发布版本不一致的问题）。

## 坏了要回退 · 怎么恢复

- **双击**：`维护-回滚到最近稳定版本`。
- **命令**（在仓库根目录）：
```bash
git log --oneline -10          # 看最近提交，找改坏之前那个的哈希
git stash                      # 先把当前没提交的改动收起来（保险）
git checkout <好的那个哈希> -- <被改坏的文件>   # 只回退某个文件
# 或整棵树回到某个提交（谨慎，会丢之后的改动）：
# git reset --hard <好的那个哈希>
```
- **依赖坏了**：双击 `维护-一键重装依赖`。
- **完全懵**：`docs/传承/紧急恢复卡片.md` + `维护-导出诊断日志`（把诊断发给会帮你的人/AI）。

## 绝对不要做的事

- ❌ 不要让 AI 用 `git commit --no-verify` 跳过守卫——守卫红=有真问题。
- ❌ 不要在没备份的情况下让 AI 做"大重构/批量改"。先双击 `维护-备份关键数据配置`。
- ❌ 不要相信"隔离测试全绿=功能生效"——问它接到哪个真实入口了（见排错速查红线）。
- ❌ 不要在 `cache/` `data/` `.tmp/` `.khy/` 里找源码或删东西——那是运行时产物，源码看 `.ai/MAP.md`。
